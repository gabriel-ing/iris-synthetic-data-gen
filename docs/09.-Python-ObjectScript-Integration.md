# Python and ObjectScript Integration in SyntheticDataGen

## Current Pattern

This repository uses one shared ObjectScript loader plus copied domain asset trees instead of compiling every domain class up front.

1. `module.xml` copies `requirements.txt` and the `python/` and `ObjectScript/` trees for `FinancialServices`, `SupplyChain`, `Retail`, and `ThemePark` into `${libdir}SyntheticDataGen/`.
2. Activation calls `SyntheticDataGen.DataLoader.PersistInstallRoot()`.
3. `DataLoader` stores the resolved install root in `^SyntheticDataGen("InstallRoot")`.
4. `EnsureDatasetClasses(dataset)` lazy-loads the installed ObjectScript source directory only for the requested domain.
5. `LoadData(dataset, scaleFactor="", configPath="", clearExisting=0)`:
   - resolves the installed domain Python path
   - inserts that path into `sys.path`
   - removes cached `DataGen` and `DataGen.*` modules
   - imports `DataGen.main_iris`
   - calls the requested domain's embedded-Python load function
6. `DeleteDataset(dataset, deleteClasses=1)` clears rows and optionally deletes the compiled dataset package.

## Installed Layout

After install, the loader expects these directories under `${libdir}SyntheticDataGen/`:

- `FinancialServices/ObjectScript/`
- `FinancialServices/python/`
- `SupplyChain/ObjectScript/`
- `SupplyChain/python/`
- `Retail/ObjectScript/`
- `Retail/python/`
- `ThemePark/ObjectScript/`
- `ThemePark/python/`

## Why This Repo Uses This Shape

- Install time stays short because only `SyntheticDataGen.DataLoader` is compiled immediately.
- Each domain compiles only when requested.
- The install root is decoupled from the workspace path.
- One shared ObjectScript entrypoint can manage all four domains.
- Shared `DataGen` package names across domains are safe because cached modules are cleared before each load.

## Current Sentinel Classes

- `FinancialServices` -> `Finance.Customers`
- `SupplyChain` -> `SupplyChain.DimCustomer`
- `Retail` -> `Retail.Stores`
- `ThemePark` -> `ThemePark.Parks`

## Current DataLoader API

- `PersistInstallRoot()`
- `DefaultInstallRoot()`
- `SetInstallRoot(path)`
- `GetInstallRoot()`
- `EnsureDatasetClasses(dataset)`
- `DeleteDataset(dataset, deleteClasses=1)`
- `LoadData(dataset, scaleFactor="", configPath="", clearExisting=0)`

Important positional order for `LoadData()`:
1. dataset
2. scale factor
3. config path
4. clear-existing flag

## Example IRIS Usage

```objectscript
zpm "install SyntheticDataGen"

do ##class(SyntheticDataGen.DataLoader).EnsureDatasetClasses("SupplyChain")
do ##class(SyntheticDataGen.DataLoader).LoadData("SupplyChain",2)
do ##class(SyntheticDataGen.DataLoader).DeleteDataset("SupplyChain",0)
do ##class(SyntheticDataGen.DataLoader).LoadData("ThemePark")
```

## Reusable Pattern for Other Packages

If you want to reuse this approach in another mixed Python/ObjectScript package:

- copy Python and ObjectScript assets into a stable install directory such as `${libdir}<ModuleName>/`
- persist that install path in ObjectScript during install or activation
- pick one sentinel class per optional domain or feature set
- clear cached Python modules before importing another installed asset tree that shares a top-level package name
- keep dataset cleanup order explicit so persistent-reference deletes are deterministic
